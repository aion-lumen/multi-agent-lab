## 1. `TELEGRAM_BOT_TOKEN`-Hardcoding blockiert Multi-Bot-Setups

**Symptom:**
- `~/.hermes/hermes-agent/hermes_cli/config.py` liest fest `get_env_value('TELEGRAM_BOT_TOKEN')`
- `~/.hermes/hermes-agent/hermes_cli/dump.py` mappt `"telegram": "TELEGRAM_BOT_TOKEN"`
- Folge: `hermes kanban notify-subscribe --platform telegram` nutzt immer denselben Bot — nur ein Telegram-Bot pro Hermes-Installation via eingebautem Notify möglich.

**Use-Case (real):**
Multi-Bot-Setups sind verbreitet:
- Domain-spezifischer Bot (z.B. `TELEGRAM_BOT_TOKEN` für ImmoAlert/Council, schickt Listings an User)
- System-spezifischer Bot (z.B. `HERMES_WATCHDOG_BOT_TOKEN` für Aion-Lumen-Multi-Agent, schickt Eskalationen an User)

Brand-Hygiene zwischen Domain- und System-Notifications ist wichtig: Eskalations-Pings sollen nicht im selben Chat-Stream wie produktive Listings landen.

**Lösungsvorschlag:**
Fallback-Chain in `config.py` einführen:
1. Per-Profile-Token aus `config.yaml` (höchste Priorität — `profiles.<name>.notification_target.bot_token_env`)
2. `HERMES_TELEGRAM_BOT_TOKEN` (Hermes-spezifisch, klarer Name)
3. `TELEGRAM_BOT_TOKEN` (Backwards-Compat, niedrigste Priorität)

Aufwand: ~30 Zeilen Code, abwärtskompatibel, betrifft `config.py` + ggf. `dump.py`.

**Workaround heute:**
Eigener Polling-Service `~/Projects/aion-lumen/multi-agent/scripts/watchdog_bridge.py`. Liest Kanban-DBs read-only, sendet via separatem Token. Robust, aber Doppelimplementierung.

## 2. `hermes kanban block` — `reason` ist positional, nicht `--reason`

**Symptom:**
Diverse Architektur-Specs und Beispiele in der Form `hermes kanban block <task_id> --reason "..."`. Real ist `reason` ein **positional argument** ohne Flag.

**Beispiel:**
```bash
# Funktioniert NICHT (fehler: unrecognized arguments: --reason ...):
hermes kanban block t_abc --reason "Smoke-Test"

# Funktioniert:
hermes kanban block t_abc "Smoke-Test"
```

**`--help`-Output bestätigt:**
```
usage: hermes kanban block [-h] [--ids IDS [IDS ...]] task_id [reason ...]

positional arguments:
  task_id
  reason               Reason (also appended as a comment)
```

**Lösungsvorschlag:**
Optional `--reason` als Alias akzeptieren (argparse-mäßig kein Aufwand), oder Doku eindeutig machen.

## 3. `hermes kanban create` akzeptiert kein `--tag`

**Symptom:**
Architektur- und Onboarding-Doku referenziert Tags zur Filter-/Routing-Logik. Aber `hermes kanban create --tag domain:life-mail --tag phase:pilot ...` funktioniert nicht — `--tag` ist kein gültiges Argument.

**`--help` zeigt verfügbar:**
`--body, --assignee, --parent, --workspace, --tenant, --priority, --triage, --idempotency-key, --max-runtime, --created-by, --skill, --max-retries, --json` — kein `--tag`.

**Use-Case (real):**
Tag-Filterung von Tasks ist Standard-Workflow für Multi-Domain-Setups (life-mail, council, folio in einem Hermes-Setup). Beispiel:
```bash
hermes kanban list --filter "tag=phase:pilot"
```

**Workaround heute:**
Tags in den `--body` als Markdown-Zeile einbetten (z.B. `Tags: phase:pilot, domain:life-mail`). Maschinell schlecht filterbar.

**Lösungsvorschlag:**
- `--tag` (multiple-mal) als Argument bei `create`
- `--filter "tag=..."` bei `list`
- Eigene `task_tags`-Tabelle oder JSON-Spalte in `tasks`

## 4. `notify-subscribe` ist per-Task, kein globaler Wildcard

**Symptom:**
`hermes kanban notify-subscribe` braucht eine `task_id` als positional. Es gibt keinen Mechanismus „abonniere alle Eskalations-Events des aktiven Boards (oder: aller Boards) auf Event-Type-Basis".

**`--help`:**
```
usage: hermes kanban notify-subscribe [-h] --platform PLATFORM
                                      --chat-id CHAT_ID [--thread-id THREAD_ID]
                                      [--user-id USER_ID]
                                      task_id
```

**Use-Case (real):**
Watchdog-Pattern (Standard-Pattern in Multi-Agent-Stacks): User wird bei jeder finalen Eskalation gepingt — unabhängig davon, ob die Task bekannt ist, als sie eskaliert wird. Per-Task-Subscription erfordert, dass die Subscription bei jedem Task-Spawn hinzugefügt wird (zusätzliche Logik in jedem Worker-Skill).

**Lösungsvorschlag:**
Globale Subscriptions auf Event-Type + optionaler Filter:
```bash
hermes kanban notify-subscribe \
  --platform telegram \
  --chat-id "$CHAT" \
  --on-events blocked,gave_up,crashed \
  --filter "assignee=user" \
  --board-glob "*"
```

**Workaround heute:**
Eigener Polling-Service (siehe Befund 1). Liest `task_events` aller Boards, filtert nach `kind` und `assignee`.

## Zusatz: Schema-Diskrepanz Doku ↔ Real (klein, aber Aufwand-relevant)

**Symptom:**
Verschiedene Architektur- und Skizzen-Beispiele nennen `task_events.event_type` und `task_events.payload_json`. Real-Schema:

```sql
CREATE TABLE task_events (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id    TEXT NOT NULL,
    run_id     INTEGER,
    kind       TEXT NOT NULL,
    payload    TEXT,
    created_at INTEGER NOT NULL
);
```

**Spalten heißen `kind` (nicht `event_type`) und `payload` (nicht `payload_json`).**

Implikation für Bridge-Implementierung: SQL-Queries müssen angepasst werden. Klein, aber Stolperfalle bei direkter Übernahme einer Skizze. Schema-Doku in den Hermes-User-Guide zu übernehmen wäre hilfreich.

### 6. `profiles:` in config.yaml hat keinen Effekt auf Dispatcher

**Symptom:**
Wir haben in Phase 1 fünf Profile (architect/validator/executor/librarian/user) als Top-Level `profiles:`-Block in `~/.hermes/config.yaml` deklariert, mit `base_url`, `model`, `temperature` etc. Erwartung: `hermes kanban dispatch` würde Tasks mit `assignee=executor` an einen LM-Studio-`gpt-oss-20b`-Worker spawnen.

Real: Dispatcher überspringt sie mit `Skipped (non-spawnable assignee — terminal lane, OK)`. `hermes kanban assignees` zeigt `executor: ON DISK = no`. `hermes profile list` zeigt nur `default`.

**Diagnose:** Hermes hat zwei separate Profil-Konzepte:
1. **`hermes profile create <name>`** — legt physisches Profil-Verzeichnis `~/.hermes/<name>/` an mit eigener config, .env, SOUL.md, Skills. Diese sind „on disk" und spawnable.
2. **Top-Level `profiles:` in config.yaml** — semantisch unklar, hat im Dispatcher anscheinend keine Wirkung. Möglicherweise eine Doku/Forward-Compat-Sektion, die noch nicht implementiert ist.

Die Architektur-Synthese (`architecture-2026-05-08.md` Sektion 5) behandelt beide als dasselbe — falsch.

**Lösungsvorschlag:**
Eine der beiden Optionen:
- (a) `profiles:` in config.yaml als first-class registrieren: Dispatcher liest sie, behandelt sie als spawnable, legt Profile-Verzeichnisse on-the-fly an.
- (b) Klare Doku, dass `profiles:` heute keinen Effekt hat und Multi-Profile via `hermes profile create` angelegt werden müssen.

**Workaround heute:**
Eigener Worker-Loop `pilot_worker.py` claimt Tasks via CLI und ruft LM-Studio direkt auf.

### 7. `hermes kanban unblock <id1> <id2> ...` failed mit Multi-IDs

**Symptom:**
`unblock --help` deklariert `task_ids [task_ids ...]`. Echter Aufruf mit mehreren IDs:
```
hermes kanban unblock t_8fffcb2f t_23557dc9 t_3c014179 ...
→ cannot unblock t_8fffcb2f t_23557dc9 t_3c014179 ... (not blocked?)
```

Hermes interpretiert die Liste als ein Subjekt; nur einzelne IDs funktionieren.

**Workaround:** `xargs -n1 -I{} hermes kanban unblock {}` zur Bulk-Bearbeitung.

**Lösungsvorschlag:** argparse-Schema oder Parsing fixen, sodass alle positional args als IDs behandelt werden.

### 8. `hermes kanban reassign --to <name>` für nicht-existente Profile

**Symptom:**
Worker versuchte `hermes kanban reassign <id> --to user`, da die Architektur ein Pseudo-Profil `user` vorsieht. `user` ist aber kein on-disk-Profil. Reassign scheitert silent (oder verhält sich anders), Folge: Eskalations-Tasks verbleiben mit `assignee=executor`, bekommen aber `block`. Die Bridge filtert `assignee=user` und sieht damit keine User-Eskalationen. Pilot lief ohne Bridge-Notifications, obwohl 5 Tasks geblockt wurden.

**Lösungsvorschlag:**
- `reassign` mit explizitem Fehler bei unbekanntem Profil
- Oder: First-Class-Support für „virtual profiles" (Pseudo-Profile ohne Worker), die als Eskalations-Marker dienen — passt zur Architektur-Vision

**Workaround heute:**
Bridge-Filter erweitern, sodass auch `assignee != user`-Eskalationen detektiert werden, oder Eskalations-Comment-Pattern im Body matchen.

### 9. LM Studio + `response_format: json_object` inkompatibel

(Externe Notiz, keine Hermes-Schwäche, aber für künftige Skill-Implementierungen relevant.)

LM Studio's gpt-oss-Adapter akzeptiert `response_format.type` nur als `json_schema` oder `text` — `json_object` (OpenAI-Standard) wirft 400. Ohne `response_format` muss man via System-Prompt erzwingen, oder `json_schema` mit explizitem Schema setzen.

### 10. gpt-oss-20b ist über-konfident bei Email-Klassifikation

**Befund:**
97 Worker-Outputs mit Confidence-Wert. **Median 0.9, kein einziger Wert unter 0.7.** Damit triggert keine Tasks die Eskalations-Schwelle (confidence<0.7) → 0 Validator-Eskalationen, 0 User-Eskalationen.

**Implikation:** Das Architektur-Modell (Executor → Validator → User-Eskalation bei Unsicherheit) ist tot, solange das Executor-Modell selbstbewusst antwortet. Mehrere Möglichkeiten:
- (a) Confidence-Schwelle nach oben justieren (z.B. 0.95)
- (b) Zwei Worker-Calls per Task mit anderen Modellen, Disagreement-Detection
- (c) Hartcodierte Regeln neben Confidence (z.B. Sender-Domain-Listen, Content-Length-Anomalien)

**Relevanz für Phase 4:** ohne (b) oder (c) wird Validator-Profil im Vollerstlauf nicht genutzt — und damit die Multi-Agent-Architektur entartet zu Single-Agent.

### 13. Hermes-Default-Worker postet keinen LLM-Output als Comment

**Symptom (kritisch):** Bei `hermes kanban dispatch` für die neuen Profile passierte:
- Architect-Probe: 4× Crash (run #1–#4 alle `crashed`/`pid not alive`), 5. Versuch `completed` mit `result_len: 0` und nur einem Summary-String. Kein Comment, kein structured Output.
- Executor/Librarian/Validator-Probes: Worker exited mit `protocol_violation`: „worker exited cleanly (rc=0) without calling kanban_complete or kanban_block".
- Die 3 wurden nach 1 Failure auto-blocked (`effective_limit: 1, limit_source: dispatcher`).

**Diagnose:** Der eingebaute kanban-worker-Skill ruft das Modell auf, **erfasst aber den Output nicht zuverlässig** als Comment. Bei der gleichen Hermes-Version landet bei manchen Modellen das Result, bei anderen nicht. Vermutlich kontinuierlicher Reasoning-Stream-Bug bei Modellen ohne klares „end-of-content"-Token.

**Lösungsvorschlag:**
- Default-Worker robuster: bei rc=0 ohne complete/block → automatisch als Comment posten was im stdout/transcript stand, dann complete
- Oder: Doku-Hinweis wie eigene Worker-Skills strukturierten Output an Hermes liefern

**Workaround heute:** `probe_worker.py` schreibt direkt Comments + `kanban complete` via CLI. Phase 3.5b/4 Production-Worker wird auf diesem Pattern aufbauen.

### 14. `dispatcher.failure_limit=1` verhindert Modell-Loading-Retry

**Symptom:** Beim ersten Spawn musste LM Studio Modell-X laden (~10-30s). Während Loading ist der Worker nicht responsiv, Dispatcher reklassifiziert das als „crashed" und blockt den Task nach 1 Versuch.

Architect schaffte es nach 4 Crashes (run-IDs 1, 2, 3, 4 alle crashed, run #5 endlich completed) — möglicherweise weil Architect-Profil eine andere `failure_limit` hat (Default vs explizit).

**Lösungsvorschlag:**
- Erst-Spawn-Toleranz: Modell-Loading-Window (z.B. 30s) wird nicht als crash gezählt
- Oder: per-Profil `failure_limit` einfach erhöhen (Doku, wie?)

**Workaround heute:** `--failure-limit N` flag bei `hermes kanban dispatch` (in Phase 3 schon entdeckt — default 2)

### 15. LM Studio gpt-oss `reasoning_effort: "low"` schluckt Tokens

**Symptom:** gpt-oss-20b mit `reasoning_effort: "low"` produziert oft leeren `content`-String, weil Tokens für internen Reasoning-Trace verwendet werden. Bei längeren System-Prompts (z.B. SOUL.md eingebettet) verstärkt sich das.

Mit `reasoning_effort: "minimal"` antwortet gpt-oss in 3s mit 38–270 Zeichen sauberem JSON. Reasoning-Trace landet im separaten `reasoning`-Feld der OpenAI-API-Response (LM Studio's Extension).

**Lösungsvorschlag:**
Für gpt-oss als Executor-Modell: `reasoning_effort: "minimal"` als Default. `low` ist nur für längere Begründungs-Tasks geeignet.

**Hermes-relevant:** wenn Hermes-Worker-Skill `reasoning_effort` nicht explizit setzt, hängt es vom Profil-Default ab. Phase-3.5a-config.yaml hat `reasoning_effort: low` für executor — sollte für Phase 4 auf `minimal` geändert werden.

### 18. Telegram-Bridge sendet pro Eskalation 3 Pings (vor Dedup-Fix)

**Symptom:** Pro Disagreement-Task kamen 3 Telegram-Pings:
- 1× `gave_up` (Hermes-Default-Worker protocol_violation, Befund 13)
- 1× `blocked` (unser Worker via `hermes kanban block`)
- 1× `blocked` (Re-Run-Cycle-Block falls User unblockt aber Re-Disagreement)

**Fix:** Bridge-State-File erweitert um `notified_tasks`-Set pro Board. Bei jedem neuen Eskalations-Event prüft die Bridge ob `task_id` schon notified — wenn ja, skip. Re-Eskalation würde manuelles Reset des Sets erfordern.

**Schema-Migration:** legacy `{board: int}` → `{board: {last_event_id, notified_tasks}}`, abwärtskompatibel.

### 19. User-Override-Format: Worker-Pattern „Final: <cat>" zu strikt

**Symptom:** User antwortete im Terminal mit nacktem Kategoriewort („geschaeftspost", „spam") — der Worker erkannte das nicht und re-prozessierte die Task.

**Fix:** `parse_user_final` akzeptiert jetzt:
- `Final: <cat>` / `Final = <cat>` (originales Pattern)
- nackte Kategorie als gesamter Comment-Body
- erste matchende Kategorie irgendwo im Body

Plus: `wait_for_unblock` filtert nicht mehr auf „Final:"-Substring, sondern nimmt den letzten Comment mit `author NOT IN {production_worker, executor, ...}`.

Plus: `find_existing_user_override` als Pre-Check — bevor Worker LLM-Calls macht, prüft er ob ein User-Override-Comment bereits existiert (z.B. von vorherigem Block-Cycle). Wenn ja: complete ohne LLM.

### 20. Body-Parsing fragil bei nicht-Standard-Markdown

1. `**Sender:**` (markdown bold)
2. `Sender:` (plain line)
3. `From:` (raw header)

## Phase-3.5c-Befunde (2026-05-12)

### 25. Schema-v2.1-Prompt-Hint reduzierte LLM-Disagreement-Rate signifikant

**Befund:** Mit den 4 Tie-Breaker-Regeln direkt im `PROMPT_TPL` (max 10 Zeilen Hint) sank die Disagreement-Rate im LLM-Pfad von 31% (Phase 3.5b: 8/26) auf 8% (Phase 3.5c: 2/26). Executor + Validator einigten sich in 21 von 26 LLM-Pfad-Tasks (81%).

**Implikation:** Schema-Doku im Prompt > Schema-Doku nur als externe Referenz. Hermes-Skill-Definitionen sollten Tie-Breaker-Regeln als kompaktes Pattern direkt im Prompt-Template vorsehen, nicht nur als Backing-Markdown.

### 26. `ambiguous_senders`-Routing erlaubt Subject-Aware-Override ohne Schema-Aufweichung

**Befund (positiv):** Die neue `ambiguous_senders`-Liste mit Subject-Score-Routing löst das Phase-3.5b-„heuristik_zu_breit"-Problem (Polymarket): Marketing-Subject → werbung, Service-Subject → route_to_llm. Im Re-Run #3 wurde Polymarket je nach Subject korrekt klassifiziert; bei Subject-Score=0 routet die Heuristik an LLM weiter, anstatt eine harte Klassifikation zu erzwingen.

**Implikation:** Wenn Hermes-Skill-Definitionen pre-classification (Routing) vorsehen, sollten 3-Wege-Outcomes (`match` / `match-with-subject-check` / `pass-to-llm`) als First-Class-Konstrukt verfügbar sein, nicht nur binär (hit/miss).

### 27. `marketing-patterns.json` als externes Lernspeicher-File funktioniert

**Befund:** Eine kleine JSON-Datei (17 Subject-Patterns + 8 Emojis) als Backing-Store für `subject_marketing_score()` ist robust + idempotent + per cron-job erweiterbar. Pattern-Score≥1 vs ≥2 (für ambiguous bzw. service-domain-override) liefert in der Praxis sinnvolle Trennung.

**Implikation:** Hermes-Boards profitieren von einem konventionalisierten „state-store pro Board" (z.B. `~/.hermes/kanban/boards/<slug>/state/`) für persistente Worker-Lerndaten ausserhalb von SQLite.

### 28. GT-Genauigkeit +20 Prozentpunkte ohne Vollerstlauf, nur durch Lern-Module

**Befund:** Phase 3.5b raw 9/15 = 60% → Phase 3.5c raw 12/15 = 80%. Verbliebene 3 Mismatches sind alle reine Schema-Konflikte (2× NGO mit GT=`newsletter_business` v1-Label, 1× Makler-Antwort GT=privat vs Schema-v2.1-`geschaeftspost`). Bereinigt: 15/15 = 100%.

**Implikation:** Schon der Lern-Schritt aus 15 Diagnose-Mails (Phase 3.5b) genügt, um 2 echte Heuristik-Mismatches dauerhaft zu eliminieren (Andreas Heinze → privat, Polymarket → werbung). Phase 4 (Vollerstlauf 3000 Mails) wird die Lerntabelle weiter anreichern.

### 29. Schema-Failures durch reasoning_effort=minimal bleiben bei ~10% der LLM-Tasks

**Befund:** 3 von 26 LLM-Pfad-Tasks (11.5%) lieferten kein gültiges JSON nach Re-Prompt — alle drei landeten in User-Eskalation. Konsistent mit Phase-3.5b-Beobachtung.

**Implikation:** gpt-oss-20b mit `reasoning_effort=minimal` ist schnell genug für Pilot, hat aber ~10% Schema-Drift. Phase 4 sollte evaluieren: reasoning_effort=low + structured output (z.B. via `response_format: json_object` falls LM Studio das stabilisiert) vs JSON-Repair-Library.

### 30. Custom Worker-Lane-Pattern hat zweite Iteration ohne Bruch überlebt

**Befund:** Das `production_worker.py`-Pattern (Phase 3.5b) konnte für Phase 3.5c mit nur 4 minimalen Änderungen (Import-Switch, Subject-Aware-Aufruf, Schema-v2.1-Prompt-Hint, Lane-Docstring) erweitert werden — Worker-Output-Schema, Hermes-CLI-Integration, Bridge-Filter blieben unverändert. Re-Run #3 lief auf Anhieb stabil (nach venv-Python-Fix).

**Implikation Issue #20157:** Das Pattern „Custom Worker mit Heuristik vor LLM, Hermes als Coordination-Layer" ist tragfähig genug für eine zweite Lern-Iteration. Eine native Lane-Registration-API in Hermes sollte dieses Muster (Pre-Filter + LLM-Cascade + User-Escalation-via-block) als Reference-Case adressieren.
